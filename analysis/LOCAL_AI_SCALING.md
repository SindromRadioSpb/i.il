## 3. Главная архитектурная задача: как соединить локальный GPU с Cloudflare

Cloudflare Worker работает на серверах в датацентрах — он не знает о твоём компьютере.
Чтобы Worker мог использовать локальный Ollama, нужен **мост**.

```
[Cloudflare Worker] ──HTTP──► [публичный endpoint] ──► [Ollama :11434]
                                    ↑
                             один из трёх вариантов
```

---

## 4. Три варианта архитектуры

---

### ВАРИАНТ 1 — «Туннель» (минимальные изменения, максимально быстрый старт)

**Что делаем:** Пробрасываем локальный Ollama в интернет через бесплатный Cloudflare Tunnel.
Worker вместо вызова `https://generativelanguage.googleapis.com` вызывает `https://my-ollama.cfargotunnel.com`.

```
[Cron 3×/день]
     │
     ▼
[Cloudflare Worker] ──► [Cloudflare Tunnel] ──► [Ollama :11434 на RTX 3070]
     │                                                  │
     ▼                                          116 t/s, 3s/резюме
[D1 Database]
     │
     ▼
[Cloudflare Pages / Facebook]
```

**Плюсы:**
- Минимум кода: меняем только провайдер в `provider_chain.ts`
- Бесплатно (Cloudflare Tunnel free tier)
- Не нужна новая инфраструктура
- Деплой за ~2 часа

**Минусы:**
- ПК должен быть включён в 05:00 / 11:00 / 18:00 UTC
- Всё ещё ограничен 5 резюме × 3 запуска = 15/день (бюджет Worker 25 сек)
- Холодный старт Ollama: 37 секунд → нужно держать модель прогретой
- Туннель может нестабильно работать ночью

**Требует:**
- `cloudflared` установить на Windows
- Новый провайдер `OllamaProvider` в `summary/providers/`
- Env var `OLLAMA_BASE_URL` (секрет) → URL туннеля

**Производительность:** 15 резюме/день → без изменений (бутылочное горлышко — бюджет Worker)

---

### ВАРИАНТ 2 — «Локальный оркестратор» (рекомендуемый: баланс усилий и результата)

**Что делаем:** Worker занимается только тем, что умеет лучше всего — RSS-ингест, кластеризация, хранение в D1, публичный API. Всю тяжёлую AI-работу делает **локальный Node.js-сервис** на твоей машине.

```
[Cloudflare Worker, cron 3×/день]
     │ RSS fetch → cluster → D1
     ▼
[D1 Database] ◄──────────────────────┐
     │                                │  UPDATE stories SET summary_ru=...
     │ (D1 REST API)                  │
     ▼                                │
[Локальный оркестратор, Windows]  ───┘
     │  Task Scheduler каждые 10 мин
     │
     ├──► [Ollama qwen2.5:7b] ── 116 t/s ──► генерация резюме
     ├──► [Ollama LLaVA/moondream] ──────► описание изображений
     └──► [Facebook Graph API] ──────────► публикация постов с картинками
```

**Плюсы:**
- ПК обрабатывает столько историй, сколько физически успевает
- Нет 25-секундного бюджета Worker — можно делать 50-200 публикаций/день
- Можно добавить второй multimodal-провайдер для изображений
- Гибкость: запускай когда ПК включён, пропускай когда выключен (backlog catchup)
- D1 имеет REST API → читаем/пишем напрямую с ноутбука без изменений Worker

**Минусы:**
- Нужно написать локальный сервис (~200 строк TypeScript/Node.js)
- ПК должен быть включён для обработки (но не для публикации — это асинхронно)
- D1 REST API нужен токен доступа Cloudflare

**Производительность:**
- Каждые 10 минут: batch из 10 историй × 3 секунды = **30 секунд на batch**
- За час: **60 историй**
- За день при 8 часах работы ПК: **~480 историй** → текущий backlog (245) — за один вечер

---

### ВАРИАНТ 3 — «Полный локальный стек» (максимальная мощность, профессиональный уровень)

**Что делаем:** Worker остаётся только для **публичного API** (лента, страница истории) и хранения в D1. ВСЁ остальное переезжает на локальную машину: ингест RSS, кластеризация, суммаризация, изображения, Facebook, Telegram.

```
[Локальный оркестратор, каждые 10-15 мин]
     │
     ├── [RSS Worker] ──► 8 источников → D1 (через Cloudflare D1 REST API)
     │
     ├── [Cluster Worker] ──► новые items → stories
     │
     ├── [AI Summary Worker]
     │    ├── qwen2.5:7b → русское резюме (5 разделов)
     │    ├── qwen2.5:7b → автокатегория (politics/security/economy/...)
     │    └── qwen2.5:7b → хэштеги для FB (#Израиль #безопасность)
     │
     ├── [Image Worker]
     │    ├── Скачать og:image из оригинальной статьи
     │    └── moondream2 → описание изображения на русском (опционально)
     │
     ├── [Facebook Publisher]
     │    └── Пост = заголовок + резюме + изображение + хэштеги + ссылка
     │
     └── [Telegram Publisher] (новый канал)
          └── Сообщение = заголовок + резюме + изображение

[Cloudflare Worker] — только API serving (read-only)
[Cloudflare Pages] — только UI
[D1] — единое хранилище для обоих
```

**Плюсы:**
- Полный контроль над расписанием (каждые 15 минут вместо 3×/день)
- Неограниченное количество публикаций в день
- Изображения в постах → +100-300% охват в Facebook
- Telegram-канал как дополнительный дистрибуционный канал
- Кастомный промпт под каждую категорию новостей
- $0 стоимость AI (только электричество)

**Минусы:**
- Самый сложный в разработке (~2-3 дня)
- ПК должен работать стабильно (или Windows Service / автозапуск)
- Нужен Cloudflare API Token для D1 REST API

**Производительность:** при ПК онлайн 16 часов/день:
- **50-100 полных историй** (с резюме + изображением) в день
- Каждые 15 минут: новые истории на сайте
- До 50 постов в Facebook/день (Facebook лимит: 200/день для страниц)

### Изображения (multimodal)

| Модель | VRAM | Задача | Комментарий |
|---|---|---|---|
| **moondream2** | ~2 GB | Описание изображения | Быстро, легко — для alt-text |
| llava:7b | ~5.5 GB | Описание + VQA | Полноценный multimodal, но конкурирует с qwen за VRAM |

**Стратегия для 8 GB VRAM:**
- Держи qwen2.5:7b в памяти для текста (5.5 GB)
- Для изображений: выгружай qwen, загружай moondream2 (2 GB) → описание → возвращай qwen
- ИЛИ: если изображения не требуют описания AI — просто прикрепляй оригинал к посту (не нужен multimodal)

### Важно: изображения в Facebook без AI

Самый практичный путь к изображениям в FB-постах — **не требует multimodal**:
1. Из RSS `<enclosure url="...">` или `<media:content url="...">` берём URL картинки
2. Скачиваем изображение
3. Загружаем в FB через `POST /{page-id}/photos` с `caption=message`
4. Или используем `link_data.picture` в обычном посте — FB сам подтягивает og:image

**Это решает REQ-009 (изображения в FB) вообще без локального AI.**

## 11. Единственный риск — стабильность ПК

Вся схема работает при условии что **ПК включён**. Три сценария:

| Сценарий | Как работает система |
|---|---|
| ПК включён (игровой сеанс / работа) | Локальный оркестратор работает полным ходом |
| ПК выключен | Worker продолжает ингест и кластеризацию → черновики копятся в D1 |
| ПК снова включился | Оркестратор видит backlog → обрабатывает накопленное |

Это не проблема — это graceful degradation. Сайт всегда работает (Worker + Pages), только AI-суммаризация зависит от ПК.

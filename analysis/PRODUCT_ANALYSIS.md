## 1. Как работает система — краткий обзор

### Инфраструктура

Система построена на **Cloudflare** — глобальной сети серверов. Три компонента:

| Компонент | Что делает | URL |
|---|---|---|
| **Worker** (`iil`) | Бэкенд: сбор, перевод, хранение, API | `iil-production.sindromradiospb.workers.dev` |
| **D1** (`news_hub_prod`) | База данных SQLite | `32403483-6512-4673-aaff-a3a6e3c9aad3` |
| **Pages** (`iil-web`) | Хостинг сайта (SSR) | `iil-web.pages.dev` |

**Стоимость сейчас: ~$0/месяц** — всё укладывается в бесплатные тиры.

---

### Полный жизненный цикл одной новости

```
RSS-ленты (иврит)
    │  3 раза/день
    ▼
[1. СБОР] ────── 8 источников, 120–150 статей/запуск
    │
    ▼
[2. ДЕДУПЛИКАЦИЯ] ── SHA-256(нормализованный URL) → item_key → INSERT OR IGNORE
    │
    ▼
[3. КЛАСТЕРИЗАЦИЯ] ─ Jaccard similarity (25% порог) по токенам заголовка иврита
    │                  Окно: 24 часа назад. Результат: история = группа статей одного события
    ▼
[4. СУММАРИЗАЦИЯ] ── max 5 историй/запуск
    │                  Вход: заголовки на иврите
    │                  Провайдеры по каскаду:
    │                    1. Gemini 2.0 Flash (бесплатно до 1500/день)
    │                    2. Claude Haiku (резерв, ~$0.001/резюме)
    │                    3. Google Translate (механический)
    │                    4. Rule-based (шаблон, последний резерв)
    │                  Проверки: длина 400-700 символов, запрещённые слова,
    │                            числа из источника, атрибуция при high-risk
    ▼
[5. ПУБЛИКАЦИЯ на сайт] ── story.state → 'published' → появляется в ленте
    │
    ▼
[6. ПУБЛИКАЦИЯ в Facebook] ── max 5 постов/запуск
                               Graph API v21, Page Access Token (non-expiring)
                               Формат: 📌 Заголовок + 2 строки резюме + ссылка
```

### Структура одной новости на сайте

**Лента (`/`):** карточка = заголовок + первые 250 символов резюме + категория + дата + кол-во источников

**Страница истории (`/story/:id`):**
```
[Категория-бейдж]  [дата]

Заголовок события

Что произошло: ...
Почему важно: ...
Что дальше: ...
Источники: Ynet, Mako, Walla

[Источники со ссылками на оригинальные ивритские статьи]
[Таймлайн заголовков оригинальных статей]
```

### Финансовая модель (текущая)

| Услуга | Расход | Стоимость |
|---|---|---|
| Cloudflare Worker + D1 + Pages | всё в бесплатном тире | **$0/мес** |
| Gemini 2.0 Flash | 15 запросов/день = бесплатный лимит | **$0/мес** |
| Claude Haiku | резервный, редко | **~$0–1/мес** |
| Facebook Graph API | официальный | **$0** |

**Граница масштабирования:** увеличение суммаризации до >15 AI-вызовов/день → выход за бесплатный Gemini tier → появляются расходы (~$0.15/1000 вызовов).

## 2. Текущие ограничения

### L-001 — Примитивная кластеризация (Jaccard по токенам)
**Проблема:** Алгоритм не понимает семантику. Два теракта в Тель-Авиве могут объединиться в одну историю. Одно событие с разными формулировками может расщепиться на две.
**Влияние:** Качество группировки событий — среднее. Читатель может видеть дубликаты или неполные события.

### L-002 — Входные данные только заголовки (нет тела статьи)
**Проблема:** AI видит только 50-80 символов ивритского заголовка. Никакого контекста, деталей, цитат. Резюме строится из очень скудного материала.
**Влияние:** Качество резюме ограничено. Невозможно написать "министр сказал что...".

### L-003 — Категории не проставляются автоматически
**Проблема:** Поле `category` у большинства историй = `'other'` по умолчанию. Категории из `category_hints` источников не транслируются на уровень истории автоматически.
**Влияние:** Фильтрация по категории бессмысленна. Бейджи на карточках показывают "другое".

### L-004 — Нет изображений
**Проблема:** RSS-ленты часто содержат `<enclosure url="...">` с обложкой статьи. Система их игнорирует.
**Влияние:** Лента выглядит текстово. Facebook-посты без картинок получают меньше охвата.

### L-005 — Нет приоритизации историй
**Проблема:** Срочная новость о теракте стоит в той же очереди суммаризации (5/запуск), что и новость о спортивном матче. FIFO.
**Влияние:** Важные события могут выйти с задержкой 8–24 часа.

### L-006 — 245+ черновиков в очереди, выход по 5/запуск
**Проблема:** Накопился backlog. При 15 публикациях/день на его разгрузку уйдут недели.
**Влияние:** Читатель видит старые новости. Актуальность страдает.

### L-007 — Facebook-пост минималистичен
**Проблема:** Формат: `📌 Заголовок + 2 строки + ссылка`. Нет хэштегов, нет изображений, нет форматирования под FB-алгоритм, постинг в 05:00/11:00/18:00 UTC (не прайм-тайм для русскоязычной аудитории).
**Влияние:** Низкий органический охват, мало вовлечённости.

### L-008 — Сайт статичен, нет real-time обновлений
**Проблема:** Читатель должен вручную обновлять страницу, чтобы увидеть новые истории.
**Влияние:** Плохой UX для читателей, которые держат сайт открытым.

### L-009 — Нет поиска и фильтрации
**Проблема:** Единственная навигация — бесконечная лента + кнопка "Загрузить ещё".
**Влияние:** Невозможно найти конкретное событие. Плохая навигация.

### L-010 — Нет подписки / уведомлений
**Проблема:** Нет email-рассылки, push-уведомлений, Telegram-бота.
**Влияние:** Пользователь должен сам вспомнить зайти на сайт.

### L-011 — Тайм-бюджет 25 секунд — архитектурное ограничение
**Проблема:** Инест + суммаризация + FB в одном Workers-вызове за 25 секунд. Нельзя масштабировать количество публикаций без перехода на отдельные cron-задачи.
**Влияние:** Жёсткий потолок на 5 резюме и 5 FB-постов за запуск.

### L-012 — min_interval_sec не реализован
**Проблема:** В `registry.yaml` у каждого источника есть `min_interval_sec`, но `ingest.ts` не проверяет время последней выборки. Источники вызываются при каждом запуске.
**Влияние:** Minor — при 3 запусках/день не критично.

### L-013 — Нет системы метрик для читателей
**Проблема:** Нет счётчика просмотров, нет аналитики, нет информации о том, какие истории читают.
**Влияние:** Невозможно принимать решения на основе данных.
